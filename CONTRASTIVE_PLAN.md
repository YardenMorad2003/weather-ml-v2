# Contrastive learning model — implementation plan

Handoff doc for the next coding session. The goal is to add a learned
similarity model on top of weather-ml-v2's existing classical pipeline,
trained via contrastive learning on a synthetic dataset and fine-tuned
per user from their tournament picks.

## The plan

Two-stage architecture, both stages contrastive:

### Stage 1 — global pretraining on synthetic data

**Goal:** learn a 32-d shared embedding space where text queries and
city profiles are aligned. Trained on ~10k synthetic
`(query, positive_city, negative_city)` triples generated by an LLM.

**Encoders:**
- Text: frozen `sentence-transformers/all-MiniLM-L6-v2` (22M params,
  ships at 384-d) → trainable MLP projection head (`384 → 32`).
- City: trainable MLP `96 → 64 → 32`.
- Total trainable params: ~20K.

**Loss:** InfoNCE / contrastive cross-entropy with batch size 256,
in-batch negatives. Optionally hard negatives from the existing
weighted-Euclidean recommender's bottom-of-rank for the same query.

**Training time on a free Colab T4:** ~1 min once text embeddings are
precomputed. Iteration is essentially free.

### Stage 2 — per-user fine-tuning from tournament picks

**Goal:** personalize the metric to each user using their 20-round
tournament picks (extending the current 10-round tournament).

**Architecture:** freeze both encoders. Add a tiny user-specific
adapter — a `32×32` linear transform + bias = 1056 parameters.

```
d_user(anchor, city) = ||W_user · f(city) + b_user − anchor||
```

`W_user` initialized to identity so the fine-tuned model starts equal
to the base. Trained with **triplet loss** on the 20 user triples
`(anchor_t, picked_t, rejected_t)`, heavy L2 regularization.

**Training time:** <1 second on CPU. Runs server-side after the user
completes their 20-round tournament.

## Synthetic dataset generation (Stage 1 input)

This is the critical-path item — start here.

- Use OpenAI `gpt-4o-mini` (already wired up via `nl_parser.py`)
- Generate 10k diverse free-text queries ("somewhere like Tokyo but
  drier", "warm coastal European cities", "places like Denver but
  greener", etc.). Use a prompt that encourages variety.
- For each query, ask GPT-4 to pick top-3 best matches and 5 worst
  matches **from the canonical 230-city list**. Constrain output via
  Pydantic to avoid hallucinated city names.
- Output JSON: `[{query, positives: [city, city, city], negatives: [...]}]`
- Generation time: ~30 min – 2 hours depending on concurrency. Cost:
  a few dollars on `gpt-4o-mini`.
- **Manual spot-check** ~100 random samples before training. If the
  LLM is producing nonsense, tweak the prompt and regenerate.

Save to `backend/cache/contrastive_triples.json`. Treat it as
ground-truth supervision.

## Eval

- Hold out 10% of synthetic triples as test set.
- Metric: **recall@10** on test set. Target >70%.
- Also: compare side-by-side with the classical recommender on real
  queries from the README's example list ("Seattle but sunnier", etc.).
  Demo slide: same query → both rankings → discuss the differences.

## File conventions

Match the existing repo's patterns:

- Backend ML code → `backend/services/`. New module:
  `contrastive.py`. Don't put training scripts in `services/` — they go
  in `backend/scripts/`.
- Cache files → `backend/cache/*.npz` or `.json`. `.gitignore` already
  has selective exceptions; add new exceptions explicitly.
- Frontend → `frontend/app/<route>/page.tsx`. Client components only
  (the site is `output: "export"` — no SSR data fetching).
- Pydantic models in routers, dataclasses in services. Match the
  shape of `recommender.py` / `tournament.py`.
- Don't break the existing query / explorer / tournament tabs. The
  contrastive model should be a new option, not a replacement.

## Decisions still open

1. **Do we extend the tournament to 20 rounds always, or only in a
   "Personalized" mode toggle?** Recommend opt-in toggle so the default
   experience stays at 10 rounds.

2. **Do we wire vote logging to the DB now, or just hold sessions in
   memory?** In-memory is faster to ship; DB logging is rubric-friendlier
   ("real data infrastructure"). Recommend: do both — log to DB AND
   cache in-memory.

3. **Train MiniLM end-to-end or keep it frozen?** Frozen is the safe
   default. If time permits, train end-to-end as an ablation for the
   demo slide.

4. **Hard negatives or only in-batch negatives?** Start with in-batch,
   add hard negatives if recall@10 plateaus.

## Rubric pressure points

(From the syllabus, what graders will look for)

- **Algorithm Implementation (10%)** — at least one algorithm
  implemented from scratch, not a sklearn wrapper. Both InfoNCE and
  triplet loss qualify; both need to be hand-written in PyTorch (no
  `torch.nn.TripletMarginLoss` shortcut for the headline algo).
- **Working Demo (10%)** — must run end-to-end live. Pre-trained
  weights need to be checked into the repo (or auto-downloaded on
  startup) so graders can run without re-training.
- **Live Presentation (10%)** — every team member must explain their
  module. Architecture diagram + the InfoNCE / triplet loss equations
  on a slide are essential.
- **PCA caveat** — current PCA is `sklearn.decomposition.PCA(...)`,
  which the rubric flags as "4/10 pts, mostly a library call." Either
  rewrite in numpy SVD (~20 lines, drop-in) or designate the
  contrastive model as the from-scratch algo and accept PCA as
  background plumbing. **Recommendation: rewrite PCA in numpy** — it's
  a 45-min job and de-risks the grading. See `state.py:26` for the
  current call site.

## Demo story (for slides)

Three-slide arc:

1. **The pretrained model.** Architecture diagram: dual encoder + InfoNCE.
   Show a t-SNE / PCA of the 230 city embeddings, color-coded by

2. **Per-user fine-tuning.** Show the adapter (`W_user · f(city) + b`).

## Things to NOT do

- Don't break existing tabs (Query, Explorer, Tournament). The
  contrastive model is additive.
- Don't re-fetch the climate data; the npz at
  `backend/cache/profiles_2023-01-01_2024-12-31.npz` is the source of
  truth.
- Don't ship pre-trained weights >5 MB committed to git. Use Git LFS,
  or have the backend download on first startup, or check the size
  carefully. The city encoder MLP weights should be tiny (~30 KB).
- Don't introduce new top-level dependencies without checking
  `requirements.txt`. `torch` and `sentence-transformers` need adding;
  watch the Render free-tier image size limit.
- Don't fake the eval. If recall@10 is bad, say so and explore why.
  The rubric rewards honest analysis.
